# ADR-010 — Un bug = una hipótesis. No se mezclan hipótesis.

**Estado:** Aceptado

## Decisión

Ante cualquier reporte de bug, se aísla y confirma **una** hipótesis a la vez. No se
investigan ni corrigen múltiples hipótesis simultáneamente en una misma sesión de
debugging.

## Por qué

Mezclar hipótesis produce dos fallas típicas:

1. Arreglar algo que en realidad era código desactualizado (un ZIP viejo, caché) o un
   problema de entorno, no un bug real.
2. Introducir cambios innecesarios porque no se aisló cuál de varias causas posibles era la
   real.

## Protocolo derivado

Ver `docs/DIAGNOSTIC_PROTOCOL.md` para el proceso de 6 pasos que opera esta ADR.

## Nota de implementación relacionada

(Esto es implementación, no forma parte de la decisión de arquitectura en sí — se anota acá
solo para contexto histórico de por qué existe.)

Al aplicar este ADR sobre un bug reportado, el protocolo requiere poder identificar la build
corriendo (paso 1). Eso expuso que el proyecto no tenía ningún mecanismo de versionado. Se
agregó `BUILD_INFO.json` + un hash automático de `core/knowledge/` calculado en `app.py` al
arrancar, expuesto en `/api/build_info`, en el footer de la UI y en la consola del
navegador.
